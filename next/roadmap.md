# Praxis Roadmap

Tracked work items for the Praxis research framework. These range from near-term wins to longer-term architectural goals. Items are roughly ordered by priority, not dependency.

---

- [ ] **Scheduled sampling at training time**
      Train the model on its own predictions for a fraction of positions: instead of always feeding the ground-truth previous token, occasionally substitute the model's sampled token. This closes the train/inference distribution gap that drives exposure bias (Bengio et al. 2015). Hook in at the trainer's input-id construction step; expose the schedule (start at 0%, ramp to ~25% over training) as a CLI knob. Natural A/B against the existing `contrastive_token` loss, which targets the same failure mode from a different angle.

- [ ] **Cyclic Mixture-of-Depths routing (ping -> pang -> pong -> ping)**
      The current MoD routers route a fraction of tokens through each layer in the forward direction only. A cyclic variant would let tokens revisit earlier layers, turning the depth dimension from a chain into a graph - the substrate for "hard tokens loop back through heavy layers while easy tokens take the short path." Cheapest first probe: add a new entry to `ROUTER_REGISTRY` that, after a layer's normal selection, sends some fraction of the surviving tokens back to a randomly-chosen earlier layer. Measure whether the cyclic paths concentrate on high-entropy positions, code, or structured output.

- [x] **Region-conditional sampling parameters (low temp inside `[TOOL_CALL]`)**
      Done for the primary inference path. The `Generator` token loop (`praxis/generation/generator.py`) adds the `[TOOL_CALL]`/`[/TOOL_CALL]` tokens to the EOS list so generation halts at each boundary, flips an `in_tool_call` flag, and inside the region forces `do_sample=False` and strips `temperature`/`top_k`/`top_p`/`renormalize_logits` - greedy decoding for the JSON body, where sampling noise would break the downstream parse - then restores normal sampling after the close token. Region boundaries reuse `find_unprocessed_tool_call_ids` (`praxis/tools/tags.py`). Not done: the streaming `MonoForwardGenerator` (Ray inference) still carries a `TODO` (~line 183) - its yielding loop needs a different control-flow shape than the halt-and-resume `Generator`, so tool-call regions there still use the request's normal sampling. The broader ideas floated here (grammar-constrained decoding; raised temp in narrative / lowered in code blocks) were not pursued. Follow-up if it matters: port the deterministic-region switch into the MonoForwardGenerator stream loop.

- [ ] **Learned / dynamic per-token temperature**
      Explore having the model predict its own sampling temperature alongside its token logits, rather than treating temperature as a fixed inference-time hyperparameter. The intuition: the model is the entity best positioned to know when it is confident (use low temperature) vs. exploring (use higher temperature). Implementation sketch: a small auxiliary head off the final hidden state that emits a scalar (or short vector) controlling the softmax temperature for the *next* token's distribution, trained jointly with NLL. Related prior art: Mirostat (Basu et al. 2020) for dynamic-temperature sampling without a learned predictor, and "Hot or Cold? Adaptive Temperature Sampling for Code Generation with Large Language Models" ([Zhu et al. 2023, Peking University](https://arxiv.org/abs/2309.02772)), which adapts temperature by token type via a heuristic (high for "challenging" tokens, low for "confident" ones) rather than a learned predictor. NB: the link above resolves to the Zhu et al. paper, *not* an Apple work - the separately-recalled "Apple paper on learned per-token temperature" remains unidentified, if it exists at all. Worth pairing naturally with the region-conditional sampling item above.

- [x] **Build a shared human/AI toolbox at `tools/`**
      Create a top-level `tools/` directory for small CLI utilities both the human and the assistant can call directly, instead of asking each other to read/copy/paste data between contexts. First candidate: a metrics-API client that hits the running web app's `/api/dynamics`, `/api/metrics`, and `/api/harmonic_spectrum` endpoints and returns summary stats (latest values, recent slopes, NaN/missing counts, spectrum band concentration) so the assistant can self-diagnose training runs without the human having to relay numbers off charts. Each tool should be small, single-purpose, and run via `python tools/<name>.py`. Long-term: a place for shared tooling that lowers the cost of collaboration on every task.
      First tool shipped: `tools/fetch_metrics.py` (stdlib-only, no new deps). Pulls `/api/metrics` by default, `/api/dynamics` with `--dynamics`, defaults to the current run, supports `--run <hash>` and `--filter <regex>`. Per series it prints n, latest, first, min, max, slope-per-1k-steps (least-squares over the last 20% of points), missing count, and a 16-glyph unicode sparkline. `--json` emits the same summary in machine-readable form. Scope decisions made along the way: skipped real terminal charting (deps + hard to paste into chat), and dropped the original `/api/harmonic_spectrum` mention from the roadmap text because that endpoint no longer exists - harmonic-spectrum lives under `/api/head_snapshots` now, and its grid/snapshot shape doesn't fit the time-series summarizer; a separate `tools/fetch_snapshots.py` is the right home for it when wanted. Naming convention established: scripts under `tools/` use verb-prefixed names (`fetch_metrics.py`, not `metrics.py`) so they read naturally as actions if any of them ever get exposed to a Praxis model as a callable tool.

- [ ] **Finish retraining tokenizers at varied sizes**
      Complete the in-progress work of retraining the project's tokenizers across a range of vocabulary sizes so each model scale has a tokenizer that actually fits it, rather than reusing a single vocab across small, medium, and large configurations. Varied sizes let us study the interaction between vocab size, embedding parameter count, and downstream loss without confounding it against the tokenizer's training corpus. Tracked in [0-5788719150923125/praxis#50](https://github.com/0-5788719150923125/praxis/issues/50).

- [x] **Environment variable support for CLI arguments**
      Every argparse flag (including dynamically added ones from integrations and experiment/environment discovery) is now settable via `PRAXIS_<DEST>` env vars. `EnvVarLoader` (`praxis/cli/loaders/env_vars.py`) iterates `parser._actions` after parsing, coerces the raw string to the action's declared type (bool-ish for store*true, type callable for scalars, comma or JSON list for `nargs`/`append`/`extend`, validated against `choices`), and applies it unless the CLI flag was given explicitly. Precedence top-down: explicit CLI > active environment YAML > `PRAXIS*\*` env var > experiment YAML > argparse default.

- [x] **Relocate web assets under `praxis/web/`**
      Moved `src/web/` (JS/CSS sources + build.py), `templates/`, and `static/` image assets into `praxis/web/`. Renamed the package from `praxis.api` to `praxis.web` since it serves both the API and the dashboard frontend. The build script now lives at `praxis/web/src/build.py` and outputs to `praxis/web/static/`.

- [x] **Finish making `METRIC_CONFIGS` fully data-driven**
      Done. The frontend no longer hardcodes any chart configs. Three registries in `praxis/metrics/training_metrics.py` are the single source of truth: `TRAINING_METRIC_REGISTRY` (scalars, pre-existing), the new `COMPOSITE_METRIC_REGISTRY` (multi-expert / sampling / routing-heatmap Research charts), and the new `DYNAMICS_CHART_REGISTRY` (gradient-flow, expert grad-norm/var, task-weight, and halting families on the Dynamics tab). Each composite/family entry carries `type` + chart hints (`key_pattern`, axis labels, `stepped`, `order`, layer-toggle/legend flags). `/api/metrics` now emits `composite_registry` and `/api/dynamics` emits `chart_registry`; `charts.js` and `dynamics.js` build their configs from these and dispatch through renderer registries keyed on `type` (`METRIC_RENDERERS`, `DYNAMICS_FAMILY_RENDERERS`) instead of the old hardcoded `if (type === ...)` chains. Deleted `METRIC_CONFIGS` from `state.js` and the per-family regex/title/dispatch literals from `dynamics.js`. Dropped the dead `arch_selection` chart (its `arch/expert_N_count` pattern never matched the backend's `architecture/{arch}_usage`). Also data-drove the live dashboard's System panel (`dashboard.js`): it now renders whatever `m.info` the backend streams, in order, minus a tiny internal denylist (`debug`, `meta`) - no more hardcoded `displayKeys`. The header/footer there stay hardcoded on purpose: they're a small fixed `live_metrics.to_dict()` contract with bespoke units/labels, not a growing registry. Adding any chart is now a one-entry change in `training_metrics.py`.

- [x] **Unify metrics logging into a single pipeline**
      Closed: the premise no longer holds. The `extra_metrics` path this item asked us to build already exists and works - `MetricsLoggerCallback` forwards every `trainer.callback_metrics` key (minus `step`/`val_*`) into `MetricsLogger.log()`, which routes non-`KNOWN_METRICS` keys into `extra_metrics`. Verified against `build/runs/`: `extra_metrics` is 100% populated on real runs (e.g. b61417c11: 17,597/17,597 rows) carrying the `halting/*` and `task_weight_*` keys this item claimed weren't flowing. The three writers are not arbitrary duplication - they sit on different cadences and access points: `metrics.db` (optimizer steps, via `callback_metrics`), `dynamics.db` (sampled gradient snapshots at `on_before_optimizer_step`, needs live `.grad`), `data_metrics.db` (data-sampling events from inside the dataloader, no trainer context). A single `step`-keyed db would be mostly NULLs/collisions, so a literal "one logger + one db" merge isn't worth it. Resolved the one real redundancy along the way: `task_weight_*` was written twice (metrics.db `extra_metrics` via `get_metrics()` + dynamics.db columns via `DynamicsLoggerCallback._extract_task_weights`); dropped the dynamics.db copy and the Dynamics tab now reads task weights out of metrics.db `extra_metrics`, the same path `routing/*` and `halting/*` already use (works for both the backprop and mono-forward trainers, not just the Lightning optimizer-step hook). Tradeoff: lost the `active_task_ids` curation that hid task types with no live data, so the chart now shows all task weights. Possible small follow-ups if desired: a shared SQLite base class to dedup logger plumbing, and collapsing the `KNOWN_METRICS`/`extra_metrics` storage split into one metrics dict.

- [ ] **Propagate task-weighted loss into layer-wise training**
      The new `loss_weights` channel is plumbed through `praxis.losses.cross_entropy`, `focal`, `mile`, `stablemax`, `halo`, and `contrastive_token` via `weighted_reduce`, but `praxis/losses/layer_wise.py` and the layer-wise / MonoForward training paths still call the criterion without forwarding `loss_weights`. Update `compute_layer_wise_loss()` to accept and pass through the per-token weights, and have the trainer thread `task_type_ids` / `assistant_mask` into each layer's local objective. Until then, layer-wise runs see uniform weighting regardless of `--task-weights` or `--mask-prompts`.

- [x] **Multi-path `--data-path` support in experiment YAMLs**
      Done. Experiment YAMLs can specify `data_path` as a list (`data_path: ["../platformer", "../rift"]`) or a scalar, and the CLI accepts the same via space-separated values, repeated `--data-path` flags, or comma-separated strings. The normalization is a single generic helper, `coerce_to_list` in `praxis/utils/arrays.py`: it flattens a scalar, a comma-separated string, a JSON-array string, or a list (whose elements may themselves be comma-separated) into a flat list of stripped, non-empty strings, with `None` -> `[]`. `main.py` runs `data_path` (and `train_datasets` / `validation_datasets`, which previously had their own inline `_split_commas`) through it; `MultiDirectoryDataset` already accepted lists. The env-var loader's old `_split_list` was deleted and now delegates to the same helper, so CLI, `PRAXIS_*` env vars, and YAML all coerce identically. Merge-vs-replace is **replace**: an explicit CLI `--data-path` wins outright; otherwise the YAML value replaces the default (no list concatenation across sources). The `launch` wrapper resolves each CLI `--data-path` value (now comma-splitting too) to an absolute host path and bind-mounts it read-only into the container. Known limitation: `launch` only discovers paths passed on the CLI, so a directory `data_path` that lives **only** in an experiment YAML (e.g. `./launch compose --small-a`) is not auto-mounted - pass those paths via `--data-path` when running under Docker, or add them to the compose volumes.

- [ ] **Fix metrics loading performance for long runs**
      As training progresses, the `/api/metrics` endpoint gets progressively slower because it re-reads and re-downsamples (LTTB) the full metrics history from SQLite on every request. Investigate server-side caching of the downsampled response (invalidated on new writes), pre-materialized summary tables, or incremental approaches where only new rows since the last request need processing. The frontend's ETag/304 caching helps with repeated identical requests but does nothing for the first fetch after new data arrives.

- [ ] **Harden integration error handling**
      The integration loader (`praxis/integrations/loader.py`) and individual integrations silently swallow exceptions at every level - spec discovery, module loading, hook execution, and runtime errors all land in bare `except Exception: pass` or quiet `print` calls. This hides real bugs (e.g. a stale `from praxis import api` import went unnoticed because the Discord integration caught the `ImportError` and printed a one-liner). Integrations should fail loudly by default, distinguishing between import/setup errors (which are always bugs and should raise) and runtime errors (which may warrant graceful degradation). Consider a `critical` flag in `spec.yaml` so truly optional integrations can degrade without stopping training, while misconfigured ones surface immediately.

- [ ] **Unify registry pattern with a lightweight `Registry` class**
      Replace the ~22 ad-hoc `*_REGISTRY` dicts with a thin `Registry` subclass of `dict` that standardizes error messages, supports optional decorator-based registration, and provides a consistent typed interface. Migration should be incremental - the class is still a dict, so existing `REGISTRY.keys()` and `REGISTRY[name](config)` call sites stay unchanged. Also audit `SAMPLER_REGISTRY` (maps strings to strings) and `EMBEDDING_REGISTRY` (maps every block type to the same class) as candidates for removal.

- [ ] **Integrate Arxiv API as a training data source**
      Add an Arxiv-backed dataset that pulls papers via the public Arxiv API (`http://export.arxiv.org/api/query`) and streams abstracts (and optionally full-text PDFs, parsed to plain text) into the training pipeline. Register it as a new `type` in `DATASETS` (`praxis/data/config.py`) so it can drop into any collection alongside the existing HuggingFace and directory sources. Respect Arxiv's rate limits (~1 request / 3 seconds) with on-disk caching so re-runs do not re-hit the API, and expose category filters (e.g. `cs.LG`, `cs.CL`) as dataset config so we can target research-relevant subsets.

- [ ] **Experiment with Backpack-style multi-sense embeddings**
      Prototype a Backpack Language Model ([Hewitt et al. 2023](https://arxiv.org/abs/2305.16765)) embedding layer, where each token gets *k* learned "sense vectors" rather than a single embedding, and the model produces contextual output as a weighted sum over the senses of tokens in the sequence. The appeal is interpretability and compositionality: senses in the original paper specialized in ways that were human-legible (syntactic role, lexical content, etc.) and could be intervened on directly. Fits naturally as a new entry in `EMBEDDING_REGISTRY` so it can be swapped against the default embedding without touching the rest of the stack. Worth testing whether sense specialization survives in our small-scale regime, and whether it composes with Prismatic / per-depth bias experiments.

- [ ] **Table-free projection embeddings (SGNN / PRADO style)**
      Prototype an embedding that *computes* token vectors from a fixed hash/LSH projection of byte n-grams instead of retrieving them from a `vocab x dim` table, following SGNN ([Ravi & Kozareva 2018](https://aclanthology.org/D18-1105/)) and PRADO / pQRNN. The fixed projection carries zero learnable params and leans on Johnson-Lindenstrauss to preserve distances, with a small learned layer downstream recovering features - trading embedding params/VRAM for a bit more compute. Fits as a new `EMBEDDING_REGISTRY` entry, and pairs naturally with the byte-latent stack, which already hash-embeds n-grams.

- [ ] **Fractional-seed init sweeps for ablation sensitivity**
      Generalize the integer `--seed` to a float, where the fractional part interpolates the initial weights between two adjacent integer seeds (e.g. `3.14` blends 14% of `seed=3` weights with 86% of `seed=4`). Closest known analogue is the variation-seed / subseed-strength mechanic in Stable Diffusion, applied to noise rather than weights. The use case here is a cheap 1D walk through init space: sweep the fraction during an ablation to measure how smoothly a metric varies with init, rather than getting a single noisy point per integer seed. Note that for i.i.d. Gaussian inits, naive lerp shrinks variance at α=0.5, so the blend should rescale (or use slerp) to keep the init distribution well-behaved.

- [ ] **Explore neuro-symbolic path prediction with branchless execution**
      Prototype a small neuro-symbolic core where the "program" is an explicit symbolic state machine (registers, ops, routing table) and tiny neural networks act as differentiable predicates that steer which path executes. Borrow the branchless-programming idiom: every branch becomes a soft-weighted selection over precomputed alternatives, keeping the whole system differentiable end-to-end while retaining inspectable symbolic state. Related prior art worth skimming first: Neural Turing Machines / DNC (Graves et al.), DreamCoder (Ellis et al.), AlphaGeometry's neural-guided symbolic search. The research question is whether a model like this can exceed vanilla-transformer expressivity (TC0) at small scale without relying on chain-of-thought as the sole Turing-completeness escape hatch.

- [x] **Add Titans-style learned long-term memory module**
      Prototype a Titans ([Behrouz et al. 2024](https://arxiv.org/abs/2501.00663)) memory module: a small MLP that *learns at test time* to memorize surprising inputs via a momentum + weight-decay update on its own parameters, surfaced to the main model as either context (MAC), a gate (MAG), or a dedicated layer (MAL). Fits naturally as a new entry alongside our existing attention variants in `praxis/attention/`, and pairs interestingly with the Infini/Arc work where the [[project_arc_attention_suspect]] fix established that learnable memory init was the failure mode - Titans sidesteps that by re-initializing memory per-sequence and updating it online. Worth A/B'ing against Infini on long-context recall at small scale.
      Done for 2 of the 3 surfacings. The work lives in an in-house `praxis.memory` namespace (not `praxis/attention/` as originally sketched). A single `--memory-type` flag selects a named profile from `MEMORY_REGISTRY`; each profile bundles a surfacing strategy plus memory hyperparameters, so new variants never add CLI arguments, and `none` (the default) disables the module. Shipped: `mal` (Memory-as-Layer, a test-time-learned MLP as its own residual sub-layer) and `mag` (Memory-as-Gate, a memory branch blended with attention through a learned gate), built on the shared `NeuralMemory` core with momentum + weight-decay online updates. Plus a third profile, `mal_energy`, a detached energy-based MAL update that drops the second-order graph for much lower VRAM and pairs with the surprise-triggered event chunking item below. **Not done: MAC (Memory-as-Context).** Follow-up: add a `mac` surfacing, then run the long-context-recall A/B against Infini.

- [ ] **Auto-derive each run's math expression from its compiled architecture**
      Have model classes carry a symbolic description of their own operation (SymPy, einops-style strings, or a small custom DSL), then traverse the compiled architecture and compose those fragments into a single concise formula for the full forward pass. The output lives next to every run's logs so "the math" is always inspectable and citable alongside the loss curves. Open questions: how to handle data-dependent control flow (MoD routing, halting), and whether per-module math is declared by hand or extracted from the implementation via tracing. Related prior art worth scanning first: Penzai (Google's pedagogical NN library), Equinox/Flax module reprs, SymPy lazy simplification.

- [ ] **Procedurally generate per-run research reports (LaTeX -> PDF)**
      Compile a structured paper from each run the way a build system compiles a binary: the source is the experiment config, the compiled architecture, and the empirical results; the output is a LaTeX document (PDF-exportable, optionally via a button on the web dashboard). Explicitly *not* an LLM-written prose paper - no generated narrative, no hallucinated claims. Every report follows one fixed template and looks near-identical by design; the value is the accuracy and reproducibility of the numbers, not the wording. Readers trust it precisely because it is mechanical. Sections fill from real artifacts: the auto-derived forward-pass formula from the architecture-math item above, a config / hyperparameter table, metric curves and final values pulled from the run databases (`metrics.db`, `dynamics.db`), ablation deltas across seeds and variants, and a bibliography assembled from the prior-art citations already attached to each module / registry. Think literate-programming and reproducible-research toolchains (Quarto, knitr / R Markdown, Pweave, papermill, Distill) rather than a chatbot - reports are rendered from source and results, not authored. Open questions: where citations live as structured data (per-module BibTeX vs. a central registry), how to diff two reports for an A/B comparison, and whether rendering runs offline as a `tools/` utility or on-demand behind a dashboard export endpoint.

- [ ] **$SLEEP token: shared checkpoint store, resume from where peers left off**
      Speculative. If Praxis is a standardized framework, two peers running the same `experiment.yml` against the same data should be producing compatible checkpoints. A shared store (IPFS-pinned, or a simple HTTP index to start) would let a fresh `--alpha` run fetch a peer's checkpoint that already has 20B tokens of training behind it, instead of starting from random init. The hard parts are the equivalence check (config hash + dataset manifest + tokenizer hash + Praxis version + seed lineage) and Sybil-resistant validation that a posted checkpoint actually reached the claimed loss. The $SLEEP token would reward peers who pin checkpoints other people actually resume from. Thematic fit: caching state means the rest of the network doesn't have to wake up to recompute it. Worth shelving until the deterministic-equivalence story is solid - without that, the index is a junkyard.
      `--phi`, `--pile`, `--no-source`, and the `minimal_data` feature flag are gone; selection now flows through two CLI list flags, `--train-datasets` and `--validation-datasets`, each accepting space- or comma-separated collection names (e.g. `--train-datasets base phi`, `--train-datasets base,phi`). Collections live in `DATASET_COLLECTIONS` in `praxis/data/config.py`; individual entries in `DATASETS` carry a `type` field (`huggingface`, `directory`, `synthetic-tool-calling`) so that what used to be hard-coded branches - the `./` source tree via the old `"self"` format, the synthetic tool-calling gate keyed on `"phi" in datasets`, and the auto-`dev` swap under `minimal_data` - are now just typed dataset entries dropped into whichever collection you want. `environments/dev.yml` sets `train_datasets: [dev]` directly; the 12 experiment YAMLs that previously set `phi: true` now set `train_datasets: [base, phi]`. Individual datasets are addressable as collection members today; bare-dataset selection via the CLI (without a collection wrapper) remains a future-nice-to-have.

- [x] **Context-conditioned curvature on positional geometry ("prismatic" RoPE)**
      The speculative "prismatic" warp was preempted by CARoPE ([arxiv 2507.23083](https://arxiv.org/abs/2507.23083)), Selective RoPE ([arxiv 2511.17388](https://arxiv.org/abs/2511.17388)), and PaTH ([arxiv 2505.16381](https://arxiv.org/abs/2505.16381)) - they collectively fill the "content-dependent cumulative-rate RoPE warp" slot and its matrix generalization; GRAPE ([arxiv 2512.07805](https://arxiv.org/abs/2512.07805)) covers the formalism. FoX is score-side decay, not a position encoding.
      Shipped instead: **HoPE** ([arxiv 2410.21216](https://arxiv.org/abs/2410.21216)) at `praxis/encoding/hope.py`, with the encoding registry unified across `CausalAttention`/`InfiniAttention`/`ArcAttention` (`before_scores` / `build_score_mod` / `after_scores`). Caveat: the deleted inline `_apply_rope` paired `(Q[2i], Q[2i+1])`; the registry pairs `(Q[i], Q[i+d/2])`. Legacy `attention_type ∈ {causal, infini, arc}` + `encoding=rope` checkpoints are not bit-compatible; `modular` was already on the registry convention.

- [ ] **Learnable RoPE theta, per-depth in ArcAttention**
      The default `theta=10000` is inherited from the original RoPE paper and almost certainly wrong at our scale; nobody has empirically tuned it for tiny models. Prior art on making it learnable: [IEEE Access 2025](https://ieeexplore.ieee.org/document/11084811) reports modest-but-consistent gains with four stabilizing tricks (separate LR, layer-wise init, cosine anneal, sigmoid bound) on Tiny Shakespeare / WikiText-103 / IWSLT'14; a [Frontiers 2025 study](https://www.frontiersin.org/journals/computer-science/articles/10.3389/fcomp.2025.1626899/full) sweeps fixed theta at character-level which is directly relevant to our regime. Genuinely Praxis-shaped angle nobody else has the substrate for: **per-recurrent-depth theta in ArcAttention**, parameterized as a `nn.Embedding(depth, 1)` of log-theta deltas on top of a learned base, so each pass through the recurrence sees a different RoPE base (coarse-to-fine positional zoom across depths). Caveat: theta interacts with HoPE's cutoff (`inv_freq = 1/theta^(2i/d)`), so the simplest combination freezes HoPE's `pos_dim` at init from the base theta and lets the per-depth deltas only shift band magnitudes, not the rotated/unrotated split.
      Future work, intentionally minimal. Rather than a full vision stack, train the model to recognize text rendered as pixels and blend that signal into the standard LM objective via cross-attention. Scope deliberately small; a hook for the curvature item above if it ever wants a genuinely 2D input.

- [x] **Surprise-triggered (event-aligned) memory chunking for Titans**
      Replace the Titans memory's fixed `chunk_size` with variable-length chunks cut at surprise spikes, borrowing the event-segmentation idea from EM-LLM ([Fountas et al. 2024](https://arxiv.org/abs/2407.09450)). Both methods already key on prediction-error "surprise"; EM-LLM uses it to place episodic-event boundaries, while our memory already computes a per-chunk surprise - so the step is to let a surprise threshold (e.g. `μ + γσ` over a moving window) trigger the update boundary instead of a fixed stride. Goal: align memory writes to coherent events rather than arbitrary token windows. Pairs with the energy/Adam update work in the Titans memory module.
      Done: energy mode (`mal_energy`) runs the update on a `segment_block`=16 grid; blocks merge into events until a causal `μ + γσ` (γ=1) surprise spike or the `chunk_size` cap, whichever comes first. Boundaries reset the Adam EMAs (weights persist) via the existing parallel scan - no sequential loop on the hot path. Diagnostics: Memory Event Size (mean/min/max). Next: optional graph-theoretic boundary refinement (EM-LLM's secondary step), and per-event W0 re-freeze if the momentum-reset approximation proves too coarse.
