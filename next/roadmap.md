# Praxis Roadmap

Tracked work items for the Praxis research framework. These range from near-term wins to longer-term architectural goals. Items are roughly ordered by priority, not dependency.

---

- [ ] **Finish retraining tokenizers at varied sizes**
      Complete the in-progress work of retraining the project's tokenizers across a range of vocabulary sizes so each model scale has a tokenizer that actually fits it, rather than reusing a single vocab across small, medium, and large configurations. Varied sizes let us study the interaction between vocab size, embedding parameter count, and downstream loss without confounding it against the tokenizer's training corpus. Tracked in [0-5788719150923125/praxis#50](https://github.com/0-5788719150923125/praxis/issues/50).

- [ ] **Fix metrics loading performance for long runs**
      As training progresses, the `/api/metrics` endpoint gets progressively slower because it re-reads and re-downsamples (LTTB) the full metrics history from SQLite on every request. Investigate server-side caching of the downsampled response (invalidated on new writes), pre-materialized summary tables, or incremental approaches where only new rows since the last request need processing. The frontend's ETag/304 caching helps with repeated identical requests but does nothing for the first fetch after new data arrives.

- [ ] **Static web snapshots on Cloudflare Pages: retire the always-on public demo (2026-07-17)**
      Running `../platformer` as a public live demo has become too expensive to justify. The replacement: periodically freeze the dashboard - the built frontend plus a dump of every read-only endpoint - into a directory of static files, publish it to Cloudflare Pages (free tier: unlimited bandwidth/requests, 500 builds/month, 20k files per deployment, 25 MiB per file), and let the live server become a thing that runs only while we're actually training. We lose the interactive surfaces and keep the showcase: charts, dynamics, architecture, evolution, the KB.
      **The scaffolding mostly exists.** Three pieces are already built and were designed for exactly this seam. (1) `praxis/web/snapshots.py` - `SnapshotStore` + `SnapshotProducer` already compute the model-dependent payloads on a background cadence and stash them as plain dicts; `serve_snapshot()` reads the store and falls back to a live compute only on a cold slot. (2) `tools/render_web.py` already boots the real `APIServer` against a `_StubGenerator` with `model = None`, drives the built frontend headless, and ships a working `SOCKET_STUB_JS` (`render_web.py:357`) that fakes `window.io` so socket.io connects, goes green, and stays silent without reconnect churn. (3) Every GET route tolerates `model=None` by design (`snapshots.py:90` states the contract), and the frontend is a 34-line HTML shell whose entire UI is built by `renderAppStructure()` in `render.js` - there is no server-rendered markup to work around.
      **The five pieces of work.** (a) *Persist the model-dependent snapshots to disk.* `/api/activation_curves` and `/api/head_snapshots` are the only two endpoints that genuinely need the live model object (autograd through the activation modules; `head/criterion/encoder.dashboard_snapshots()` plus `model._rlct_landscape`). They're already snapshot-cached, so the move is to have `SnapshotProducer` also write its store to `build/runs/<hash>/snapshots/*.json` on a slow cadence - after which the export is a pure disk read and historical runs get coverage for free. (b) *The dump script* (`tools/export_snapshot.py`, per the verb_noun convention): a Flask test client over `praxis.web.app` with `snapshot_store` unset, walking a manifest of `(url, output path)`. No `APIServer`, no port, no threads. Must blank `sys.argv` during import the way `render_web.py:377` does, or `build_spec_payload` will parse the dump script's own flags into `args`. (c) *The static-mode fetch shim.* This is the one real surprise: **there is no central fetch wrapper.** `api.js` holds only about a third of the calls; 13 sites in `charts.js` / `dynamics.js` / `tabs.js` / `swarm.js` / `actions.js` call `fetch()` directly against hardcoded root-relative paths. So the interception point has to be `window.fetch` itself (or a Service Worker), mapping URL+query to a snapshot file - Pages can't vary a response by query string. Exclude the two genuinely cross-origin peer-agent calls (`charts.js:251`, `charts.js:365`). Note `/api/config` returns YAML text and `/api/paper.pdf` a blob. (d) *Gate what can't work.* POST-only surfaces need the UI greyed out, not just the endpoint stubbed: Gymnasium's `Evaluate` / `Print` / `Loop` toggles (`/messages/`, `/api/print/*`, `/api/loop/*`), the Stage tab's swarm join/heartbeat/batch, and `/api/agents` (which port-scans 2100-2120, runs `git ls-remote` per remote, and pings each - seconds of latency, nothing static about it). Git smart-HTTP (`routes/git.py`) can't be hosted on Pages at all; point the clone URL at GitHub. (e) *The offline banner.* Raise it at the notification bell, which needs a small refactor first: `mergeNotifications()` (`websocket.js:14`) is module-private and fed **exclusively** from the websocket's `metrics_snapshot` events, and those events live in an in-process `deque` on the `LiveMetrics` singleton (`live_metrics.py:54`) that is **never persisted to disk**. So the bell has no source of truth in a static build - export an explicit `notify()` helper, seed the feed with the "this is a frozen snapshot from <date>" warning, and either drop the historical events or start logging them somewhere durable.
      **Watch the refresh scheduler.** `main.js:286` is purely event-gated on the `praxis:data-invalidate` DOM event the websocket raises, and the per-tab Refresh buttons were deliberately removed (`main.js:210`). With a stubbed socket nothing auto-refreshes and there is no manual path either - which is correct for a frozen snapshot, but means the Terminal tab sits on its `"Waiting for metrics stream..."` placeholder forever unless we freeze the last `LiveMetrics.snapshot()` into the export and have the socket stub replay it once on connect. That's the cheap fix and it makes Terminal a legible static exhibit instead of a dead panel.
      **Three scope decisions that dominate the cost.** (1) *How many runs?* Per-run endpoints (`/api/spec`, `/api/metrics`, `/api/data-metrics`, `/api/dynamics`) are ~4 files each if we pin the frontend's canonical `since=0&limit=1000&downsample=lttb` query, so all 38 runs is ~150 files and maybe 50-100 MB. Multi-run compare (`runs=a,b,c`) is combinatorial and must either be dropped or merged client-side from the per-run files. Current-run-only is dramatically cheaper and probably the right v1. (2) *KB search.* `/api/kb/search` is free-text FTS5 and the single hardest endpoint: `build/kb.db` is 77 MB so the db can't ship, and covering search means a client-side index over the ~1458-item corpus. The empty-query feed dumps trivially; `/api/kb/item` is 1458 files at ~10-30 MB total. Shipping the browsable feed while disabling search-as-you-type is the honest middle. (3) *Business cards.* `/api/card/*` has an unbounded parameter space (`seed` defaults to `secrets.randbelow(2**31)`, plus 6 continuous MOD_AXES), so it's port-`render_card`-to-JS, pre-render a fixed seed set, or drop.
      **Effort:** roughly 2-4 focused days for a good v1 - call it half a day each for the disk-persisted snapshots, the dump script, and the socket/banner/notify work; a full day for the fetch shim and the gating sweep (the long tail is finding every hardcoded fetch and every control that needs greying out); half a day for the Pages deploy + a `wrangler` publish step. Scoping to the current run only and dropping KB search lands closer to 2; all-runs plus a client-side search index closer to 4. The recurring cost afterward is a periodic re-export and publish, which is a cron away.

- [ ] **Cyclic Mixture-of-Depths routing**
      The current MoD routers route a fraction of tokens through each layer in the forward direction only. A cyclic variant would let tokens revisit earlier layers, turning the depth dimension from a chain into a graph - the substrate for "hard tokens loop back through heavy layers while easy tokens take the short path." Cheapest first probe: add a new entry to `ROUTER_REGISTRY` that, after a layer's normal selection, sends some fraction of the surviving tokens back to a randomly-chosen earlier layer. Measure whether the cyclic paths concentrate on high-entropy positions, code, or structured output.

- [ ] **Integrate Arxiv API as a training data source**
      Add an Arxiv-backed dataset that pulls papers via the public Arxiv API (`http://export.arxiv.org/api/query`) and streams abstracts (and optionally full-text PDFs, parsed to plain text) into the training pipeline. Register it as a new `type` in `DATASETS` (`praxis/data/config.py`) so it can drop into any collection alongside the existing HuggingFace and directory sources. Respect Arxiv's rate limits (~1 request / 3 seconds) with on-disk caching so re-runs do not re-hit the API, and expose category filters (e.g. `cs.LG`, `cs.CL`) as dataset config so we can target research-relevant subsets.

- [ ] **Propagate task-weighted loss into layer-wise training**
      The new `loss_weights` channel is plumbed through `praxis.losses.cross_entropy`, `focal`, `mile`, `stablemax`, `halo`, and `contrastive_token` via `weighted_reduce`, but `praxis/losses/layer_wise.py` and the layer-wise / MonoForward training paths still call the criterion without forwarding `loss_weights`. Update `compute_layer_wise_loss()` to accept and pass through the per-token weights, and have the trainer thread `task_type_ids` / `assistant_mask` into each layer's local objective. Until then, layer-wise runs see uniform weighting regardless of `--task-weights` or `--mask-prompts`.

- [ ] **Table-free projection embeddings (SGNN / PRADO style)**
      Prototype an embedding that _computes_ token vectors from a fixed hash/LSH projection of byte n-grams instead of retrieving them from a `vocab x dim` table, following SGNN ([Ravi & Kozareva 2018](https://aclanthology.org/D18-1105/)) and PRADO / pQRNN. The fixed projection carries zero learnable params and leans on Johnson-Lindenstrauss to preserve distances, with a small learned layer downstream recovering features - trading embedding params/VRAM for a bit more compute. Fits as a new `EMBEDDING_REGISTRY` entry, and pairs naturally with the byte-latent stack, which already hash-embeds n-grams.

- [ ] **Auto-derive each run's math expression from its compiled architecture**
      Have model classes carry a symbolic description of their own operation (SymPy, einops-style strings, or a small custom DSL), then traverse the compiled architecture and compose those fragments into a single concise formula for the full forward pass. The output lives next to every run's logs so "the math" is always inspectable and citable alongside the loss curves. Open questions: how to handle data-dependent control flow (MoD routing, halting), and whether per-module math is declared by hand or extracted from the implementation via tracing. Related prior art worth scanning first: Penzai (Google's pedagogical NN library), Equinox/Flax module reprs, SymPy lazy simplification.

- [ ] **Fractional-seed init sweeps for ablation sensitivity**
      Generalize the integer `--seed` to a float, where the fractional part interpolates the initial weights between two adjacent integer seeds (e.g. `3.14` blends 14% of `seed=3` weights with 86% of `seed=4`). Closest known analogue is the variation-seed / subseed-strength mechanic in Stable Diffusion, applied to noise rather than weights. The use case here is a cheap 1D walk through init space: sweep the fraction during an ablation to measure how smoothly a metric varies with init, rather than getting a single noisy point per integer seed. Note that for i.i.d. Gaussian inits, naive lerp shrinks variance at α=0.5, so the blend should rescale (or use slerp) to keep the init distribution well-behaved.

- [ ] **Harden integration error handling**
      The integration loader (`praxis/integrations/loader.py`) and individual integrations silently swallow exceptions at every level - spec discovery, module loading, hook execution, and runtime errors all land in bare `except Exception: pass` or quiet `print` calls. This hides real bugs (e.g. a stale `from praxis import api` import went unnoticed because the Discord integration caught the `ImportError` and printed a one-liner). Integrations should fail loudly by default, distinguishing between import/setup errors (which are always bugs and should raise) and runtime errors (which may warrant graceful degradation). Consider a `critical` flag in `spec.yaml` so truly optional integrations can degrade without stopping training, while misconfigured ones surface immediately.

- [ ] **Experiment with Backpack-style multi-sense embeddings**
      Prototype a Backpack Language Model ([Hewitt et al. 2023](https://arxiv.org/abs/2305.16765)) embedding layer, where each token gets _k_ learned "sense vectors" rather than a single embedding, and the model produces contextual output as a weighted sum over the senses of tokens in the sequence. The appeal is interpretability and compositionality: senses in the original paper specialized in ways that were human-legible (syntactic role, lexical content, etc.) and could be intervened on directly. Fits naturally as a new entry in `EMBEDDING_REGISTRY` so it can be swapped against the default embedding without touching the rest of the stack. Worth testing whether sense specialization survives in our small-scale regime, and whether it composes with Prismatic / per-depth bias experiments.

- [ ] **Explore neuro-symbolic path prediction with branchless execution**
      Prototype a small neuro-symbolic core where the "program" is an explicit symbolic state machine (registers, ops, routing table) and tiny neural networks act as differentiable predicates that steer which path executes. Borrow the branchless-programming idiom: every branch becomes a soft-weighted selection over precomputed alternatives, keeping the whole system differentiable end-to-end while retaining inspectable symbolic state. Related prior art worth skimming first: Neural Turing Machines / DNC (Graves et al.), DreamCoder (Ellis et al.), AlphaGeometry's neural-guided symbolic search. The research question is whether a model like this can exceed vanilla-transformer expressivity (TC0) at small scale without relying on chain-of-thought as the sole Turing-completeness escape hatch.

- [ ] **(historical) CALM produces noise: audit vs shaochenze/calm (2026-05-30)**
      The port is faithful - VAE (free-bits KL), embed_proj patch input, energy head, energy_score (2·E‖X-Y‖ - E‖X-X'‖ over M=100 posterior samples), and patch-vote temperature generation all match the reference. The deviation is protocol, not math. Reference trains the autoencoder as a fully separate run (~30k steps / 15B tokens) to near-perfect reconstruction, saves it, then loads it frozen for CALM. Ours collapses both into one run with `ae_freeze_steps == warmup_steps`, so the codec freezes long before it reconstructs - stage 2 then learns to predict latents that decode to garbage. Direction: (1) verify codec fidelity first - decode posterior samples of real text and confirm it reconstructs (watch `val_codec_bpb`); (2) decouple stage 1 into its own checkpointed AE run, don't tie the freeze to LR warmup; (3) only after the AE is near-lossless, train the energy LM, and bump `energy_samples_n` (8 is a noisy repulsion estimate). Scale also matters: reference is 371M-1.8B params; tiny undertrained models may not reach fluency regardless.

- [ ] **Unify registry pattern with a lightweight `Registry` class**
      Replace the 43 ad-hoc `*_REGISTRY` dicts (currently spread across the tree) with a thin `Registry` subclass of `dict` that standardizes error messages, supports optional decorator-based registration, and provides a consistent typed interface. Migration should be incremental - the class is still a dict, so existing `REGISTRY.keys()` and `REGISTRY[name](config)` call sites stay unchanged. Also audit `SAMPLER_REGISTRY` (maps strings to strings) and `EMBEDDING_REGISTRY` (maps every block type to the same class) as candidates for removal.
      Stub landed: a `praxis.registry` namespace now exists as the home for this work, with `discover_registries()` / `count_registries()` as its first piece - an AST scan that finds every module-level `*_REGISTRY` definition (43 today). The auto-docs/README headline now reports that real number instead of a hand-maintained "~20". Renamed `NON_REGISTRY_PACKAGES` -> `INFRASTRUCTURE_PACKAGES` so no non-registry constant collides with a plain `*_REGISTRY` match. Next step toward the actual unification: define the `Registry` type in `praxis.registry` and migrate the first dict to it.

- [ ] **Preference-pair objective for hh-rlhf (and a real use for `rejected`)**
      `Anthropic/hh-rlhf` joined the `focused` collection (2026-06-10), but only SFT-style: the `chosen` transcripts are parsed into chat messages (`DataFormat.HUMAN_ASSISTANT`) and the `rejected` column is dropped on the floor. The signal that makes the dataset valuable is the _pair_. None of the existing RL policies fit: REINFORCE/GRPO need a reward channel and CoT is supervised - nothing in `RL_POLICIES_REGISTRY` consumes pairwise preferences. The natural first move is an offline preference loss, not online RL: DPO (Rafailov et al. 2023) needs a frozen reference model (awkward in a single-run framework), so reference-free variants - ORPO (odds-ratio penalty folded into SFT) or SimPO (length-normalized margin) - fit the one-model-one-run shape better. Work items: (1) a batch path that emits chosen/rejected as paired sequences sharing a prompt (the builder currently has no concept of a paired sample); (2) the loss as a registry entry alongside the existing criteria, gated to pairs-bearing batches; (3) decide its interaction with the assistant mask - preference applies to assistant turns only. Until then, hh-rlhf is just more conversation data.

- [ ] **(parked) Business-card style blending, compositionally this time**
      Two attempts at a "chimera" card (2026-06-10) that blended projection styles were built and removed same-day. What failed, and why: the blends rendered 2-3 full styles and then pruned every stroke pointwise through softmaxed/argmaxed spatial noise masks - cutting strokes at mask boundaries amputates shapes mid-curve (half a bone, a third of a lattice), and the result reads as squiggly, undefined marks that are worse versions of their discrete counterparts. Verdict from use: blend _styles_, never _renderers_. The viable rework is compositional: the unit of blending must be a whole element (one complete bone, leaf, coil, lattice), so each field would need to expose its element-placement loop (a "place one element at x,y" hook) rather than being post-processed at the artist level; a territory mask then only decides _where each style anchors its elements_, and border elements stay intact and simply overhang, the way leaves/bones already overhang the card edge. The candidate-pool/vote mechanic (counts -> dominance) was fine and can return on top of that skeleton. Until that refactor is worth doing, every card is exactly one style from `PROJECTION_REGISTRY`.

- [ ] **Learned / dynamic per-token temperature**
      Explore having the model predict its own sampling temperature alongside its token logits, rather than treating temperature as a fixed inference-time hyperparameter. The intuition: the model is the entity best positioned to know when it is confident (use low temperature) vs. exploring (use higher temperature). Implementation sketch: a small auxiliary head off the final hidden state that emits a scalar (or short vector) controlling the softmax temperature for the _next_ token's distribution, trained jointly with NLL. Related prior art: Mirostat (Basu et al. 2020) for dynamic-temperature sampling without a learned predictor, and "Hot or Cold? Adaptive Temperature Sampling for Code Generation with Large Language Models" ([Zhu et al. 2023, Peking University](https://arxiv.org/abs/2309.02772)), which adapts temperature by token type via a heuristic (high for "challenging" tokens, low for "confident" ones) rather than a learned predictor. NB: the link above resolves to the Zhu et al. paper, _not_ an Apple work - the separately-recalled "Apple paper on learned per-token temperature" remains unidentified, if it exists at all. Worth pairing naturally with the region-conditional sampling item above.

- [ ] **Adopt a proper namespaced logging interface (retire scattered `print()`)**
      The codebase emits a lot of status through bare `print()`. Move to per-subsystem loggers - `logging.getLogger("praxis.optim")`, `"praxis.data"`, `"praxis.encoder")`, etc. - so output carries a namespace + level instead of an undifferentiated stream. The plumbing already favors this: the dashboard installs a root handler (`DashboardStreamHandler`) and formats records as `name - LEVEL - message`, so namespaced loggers flow into the LOG panel with structure for free, while `print()` only survives because `start()` also redirects stdout. Wins: filterable/levelable output, per-module verbosity, clean separation of user-facing CLI text (keep as `print`) from diagnostic logging, and log lines that read like the `praxis.optim - INFO - ...` examples in the generated `terminal.webp` capture. Sketch: define a `praxis.logging` helper that hands back a namespaced logger (and sets sane defaults/levels), sweep `print()` calls that are really diagnostics over to it, and leave genuine CLI/UI output alone. Watch the interaction with the dashboard's aggressive logger reconfiguration in `TerminalDashboard.__init__` (it retrofits every existing logger) - the convention and that retrofit should be designed together.

- [ ] **Unified knowledge base: search / index / spider on the Gymnasium tab (2026-06-01)**

Terminology flag carried forward: the "differential privacy" framing is really access-control / capability-scoping + region encryption, not DP - decide which property is the actual goal.

- [ ] **Blind Election: issue-consensus as a voting primitive (2026-06-23)**
      Flagged-voice thought experiment, NOT a near-term build. Citizens vote on *issues* and the executive is *derived* from consensus across them (optionally weighting voters by the consistency of their history), rather than chosen directly - which is the **CALM patch-vote / count-based vote rebuilt as a polity**, with consistency-weighting = the elimination-tournament's reliability weighting and "blind" = the paper's refuse-to-collapse-axes decoupling. Kept deliberately *separate* from [temperament_bias_variance.md](temperament_bias_variance.md) (that note is non-prescriptive by construction; a concrete mechanism forfeits its silence). Two load-bearing caveats lead the write-up: the **discursive dilemma** (aggregating issue-by-issue can contradict aggregating the verdict - List & Pettit; must pick premise- vs conclusion-based aggregation deliberately), and **consistency-weighting IS the monoculture failure mode** (high-bias by construction, indicted by the corpus's own Wald/survivorship spine - must be bounded and paired with a variance/dissent term). Plus Arrow / Gibbard-Satterthwaite structural limits and strategic-voting adversariality. Full write-up: [blind_election.md](blind_election.md); companion to the elimination-tournament, swarm-dynamics, and pool-HEAD items.

- [ ] **Homeostatic engagement reward: the user's attention as a self-driven signal**
      A `Print` stage on the Gymnasium button row (after `Read`/`Evaluate`): the model emits a single question, the user engages, and that feeds a slowly-replenishing "energy" variable the model is optimized to keep near a setpoint - diminishing returns and a multi-hour decay, like hunger or attention (homeostatic RL, Keramati & Gutkin 2014). The satiation term is the good part: self-limiting, kills reward-spam, and mirrors the EMA-integrated-return design already in `harmonic_weight_rl`. The crux is the reward _choice_: **"questions the user answers correctly" (calibration / active learning) - NOT "maximize engagement," which is the recommender-system path to clickbait and manipulation.** Practical wall: human reward is low-volume, so scope it to tuning a small controller's narrow behavior (not a generation policy from scratch), and use a simulated user for sample density during development; an offline reward model would reintroduce proxy-gaming. Staging, smallest-first: (1) `Print` logs `(question, engaged?, correct?)` as data; (2) a homeostatic energy reward feeding a small controller, the real work being the async UI-event -> controller reward channel; (3) a `BrierLM`-style eval of the question-posing behavior. Kill criterion: if it ever optimizes attention over usefulness, stop. Full write-up: [homeostatic_engagement.md](homeostatic_engagement.md).

- [ ] **2D optimizer-trajectory visualization (real-data Rastrigin/Rosenbrock-style path)**
      pytorch-optimizer ships parametric 2D trajectory plots of each optimizer descending toy loss surfaces ([visualization gallery](https://pytorch-optimizers.readthedocs.io/en/latest/visualization/)). We want the same _picture_ but on real data: project the high-dim parameter vector to 2D and draw the optimizer's path over training. The scalar half of optimizer telemetry shipped (`praxis/metrics/optimizer.py`: the "optimizer" dynamics card group - `opt_update_rms`, `opt_update_weight_ratio`, `opt_momentum_grad_cos`, `opt_second_moment_rms`, schedule-free `opt_sf_spread` + `opt_gate_mean`); this item is the deferred non-scalar viz. Design: a _fixed random 2D projection_ of the flattened params (two frozen random unit vectors -> two dot products per snapshot, cheap and shape-stable), plotting **both** the schedule-free iterate `z` and the running average `x` as separate paths - so you watch the iterate explore while the average smooths behind it (the bias-variance story as a real-data trajectory). Reuse the existing `harmonic_curve` 2D-parametric-path renderer. Plumbing needed (the reason it's deferred): the web layer has no optimizer handle - `/api/head_snapshots` reads `model.head.dashboard_snapshots()` - so add a `/api/optimizer_snapshots` route fed by a stashed optimizer (or have the `DynamicsLogger` push trajectory points to a small ring buffer the route serves), a frontend renderer entry, and a `praxis/web/src` -> `build.py` rebuild. Per-wrapper paths (plain vs gated vs wave schedule-free) would make a clean visual ablation.

- [ ] **DreamCoder-style library learning for the CALM codec (2026-06-21)**
      Distill DreamCoder ([Ellis et al. 2020](https://arxiv.org/abs/2006.08381)), don't rebuild it (the original repo is a Singularity-based reproducibility nightmare). CALM's encoder already IS DreamCoder's "recognition model" (amortized inference of a latent); what's missing is the **library** (a compositional prior of reusable motifs) and **sleep-abstraction** (refactoring solutions into it by MDL). The tractable kernel: learn a growable, inspectable **compositional byte-motif library** by compression on a slow background cycle (reuse the spider/swarm thread infra), then encode = cheap matching against the current library - amortizing the per-chunk program search so it sidesteps DreamCoder's fatal throughput wall (billions of LM chunks vs its hundreds of tasks). Keeps a continuous latent (an embedding of the matched motifs), is never frozen, and is the "learned compositional bias" midpoint on the bias-variance codec axis (fixed harmonic = pure bias; VAE = pure variance). MDL (shortest program reproducing the data) is a principled codec objective where recon+KL is a surrogate. Honest risks: language byte-chunks may lack the program-like structure DreamCoder exploits (LOGO/lists compress; "the " doesn't); discrete library vs continuous flow target needs reconciling. **Gated on the FixedCodec ablation** ([[project_calm_fixed_codec]], `calm_byte_fixed`): if a STATIC basis is already "good enough" at our scale, a learned-library basis is the natural richer next step; if it flatlines, that question carries over. Don't build until the cheap fixed-codec result reports.

- [ ] **Multirate vote + multimodal cross-attention via sampling-as-reconstruction (2026-06-23)**
      The signal-processing rescue of "would 44K vote samples help CALM?" (no - Monte Carlo on a biased estimator). If the harmonic latent is a bandlimited signal, sample-rate is a **per-band Nyquist** quantity, not a global draw count: keep the ~500-draw budget but **stratify it across spectral bands by bandwidth** (near-DC bands want rate ≈1, high-freq bands want audio-like density) - importance sampling of the patch-vote, flat cost. **Serpent (ArcHoPE `rho_i`) is already a per-band rate dial**; the amp-modulation envelope sets each band's draw budget. Cross-attention reached *cheaply* by recognizing the **prismatic fast-weight overlay (`u_t⊗v_t`) IS linear attention** - promote the head's FiLM/adaLN conditioning to per-position selection by *querying the fast-weight state already maintained*, not bolting on softmax attention. What cross-attention is *for*: multi-view modality inputs as **fixed transforms** (FixedCodec philosophy, no learned vocoder/vision stack) - audio (TTS through standard DSP modulators; harmonic head is already a spectrum), text-on-pixel-grid sampled with **zoom-as-amplifier** foveation (reconstruct image from ~500 draws, realizing the dormant ArcHoPE "text as pixels via cross-attention" fragment), and video frames where harmonics=structure resolve like a **progressive Blender render** - sampling-as-reconstruction is one mechanism in three modality costumes. **Gated on the near-DC spectrum** (multirate buys nothing until bands carry distinct content - the amp-modulation fight); the learned zoom/foveation importance map is the unproven load-bearing part (try fixed log-polar first). Cheapest probe: confirm spectrum non-degenerate → stratified per-band vote as a gen-config A/B → single audio-spectrum cross-attention view. Full write-up: [multirate_vote.md](multirate_vote.md).

- [ ] **Edit-trajectory training: `[REMOVE]`/`[ADD]` markers as multi-timescale time**
      Teach the model the _process_ of a revision, not just its endpoints, by training the same rolling-context batch across staged passes that spell out an edit. Closest established cousins are edit-based / text-editing models (Levenshtein Transformer, LaserTagger) and diffusion LMs that learn a corruption→denoise _trajectory_; this differs in making the trajectory an explicit, marked, staged training schedule.
      The speculative core - and the reason it's hard to reason about - is that the decision space here is **timeless**. _Which_ tokens to `[REMOVE]` and _what/where_ to `[ADD]` is an order-free set of operations; the 3-step staging _injects_ a synthetic time onto it. The strangeness ("a thing like ours without time in it makes no sense") is the point: time isn't inherent to the edit, it's constructed in the data, and that's exactly the structure we'd be teaching. Honest caveats, since this is underspecified: decide the loss target at each pass (predict the marked span? the applied result?); whether it's curriculum vs. data augmentation; whether 3x-training one batch overfits; and how the 2-decision/3-step kernel generalizes to many simultaneous edits. Cheapest first probe: add `[REMOVE]`/`[ADD]` special tokens, write a synthetic generator that emits the three staged sequences from a source span (sibling to the print synthetic-samples + tasker work), train on it, and check whether the model learns to honor `[REMOVE]` and generate plausible `[ADD]` content - i.e. to _apply an edit_, not just describe one.

- [ ] **Next-Latent Prediction (NextLat): belief-state auxiliary + self-speculative decoding (2026-06-18)**
      NextLat ([arXiv 2511.05963](https://arxiv.org/abs/2511.05963), [github.com/JaydenTeoh/NextLat](https://github.com/JaydenTeoh/NextLat)) shares a name with CALM but does the opposite thing, and the contrast is the point. It is **not** a generative-latent model: the transformer still predicts discrete tokens with cross-entropy at inference, and the "latent" is just the model's own final hidden state `h_t` - no VAE, no codec, no continuous output. The method adds an auxiliary self-supervised loss that shapes those hidden states into *belief states* (sufficient statistics of history with recurrent transition dynamics, Theorem 3.2). Mechanism: a tiny residual MLP `p_ψ` (the "dynamics model") predicts the next hidden from the current hidden plus the next token's embedding, `ĥ_{t+1} = h_t + MLP(LayerNorm([emb(x_{t+1}), h_t]))`, supervised by `SmoothL1(ĥ_{t+i}, sg[h_{t+i}])` + a KL between the token distributions decoded from the true vs predicted hidden through the **frozen** `lm_head`, rolled out `d` steps (`mtp_horizon`) with teacher-forced tokens; stop-grads on the target hidden and head are the usual self-predictive anti-collapse trick. Total loss `L_ntp + λ_mse·L_next-h + λ_kl·L_kl`. Why it is easy where CALM is hard: NextLat's "next-latent prediction" is a deterministic regression to a known stop-grad target *with the next token given* - it never models a distribution or samples - whereas CALM's energy/flow head must *generate* the next latent as a sample from a learned continuous conditional with no token to lean on (exactly the failure our `probe_calm_codec.py` isolated: lossless codec, energy head stuck at acc 0). Bonus that falls out for free: because `p_ψ` rolls hidden states forward cheaply, you get self-speculative decoding - draft `d` tokens from the MLP, verify with one transformer pass - for up to **3.3x lossless** inference speedup (4.59 accepted tokens/step at d=2 on Wikipedia). Fits Praxis as a decoder/aux option: a `p_ψ` dynamics head off the trunk's final hidden, the two aux losses gated on, and the draft-and-verify loop in `Generator`. Two appeals: (1) it keeps discrete next-token CE, so fast fluency is free - the thing CALM cannot give early; (2) it is a few hundred lines, no subsystem. Caveat and the reason it is *not* a CALM substitute: it gives **none** of CALM's K-fold sequence compression - the speedup is fewer forward passes via speculation, not a shorter latent sequence. Two ideas worth lifting even if we stay on CALM: the belief-state auxiliary could shape the CALM trunk's conditioning hidden (`h_cond`) to be more self-predictable, and the speculative-decoding loop is a standalone lossless win for any Praxis token model.

That tolerance is the actual bet behind "find consistency even within the noisy train loop" - it is acceptable *because* nothing downstream depends on the preview being exact. Honest risks/open questions: (a) intra-call caching alone buys little here since `max_new_tokens` is usually 1 - the win is entirely *cross-call* prefix reuse, so the whole gain rides on the staleness bet holding; (b) interaction with recurrent depth and the Infini/Arc memory-state cache (`PraxisCache` slots keyed by `current_depth`, not just token position) complicates a purely token-prefix-keyed radix tree - the key must incorporate depth/memory state, or the encoder/recurrent paths stay excluded as they already are for `use_cache`; (c) measure first whether the stale preview visibly degrades generation quality before investing in the tree. Cheapest first probe: don't build the radix tree yet - just A/B `use_cache=True` with a *manually refreshed* persistent cache on the Terminal path (recompute the full cache every K inference calls, reuse in between) and watch both the inference-time EMA (`inference_time_ema`, already tracked) and whether the sampled text degrades; only if stale-prefix reuse proves acceptable does the radix tree (prefix sharing across the multiple temperature streams + LRU eviction) become worth the machinery.

      Salvage of the one durable idea in [HDD-RoPE](https://github.com/mikayahlevi/hdd-rope) (mikayahlevi, "high-dimensional dynamic RoPE"), evaluated and rejected *as a position encoding* but kept *as a recurrence primitive*. HDD-RoPE stacks three changes on RoPE: (1) replace the 2x2 rotation blocks with `d_c x d_c` orthogonal blocks (SO(d_c), `N = d_c(d_c-1)/2` angles each, built from a skew-symmetric matrix via a **Cayley transform** `(I-S)^-1(I+S)`); (2) make the per-step angles **data-dependent** (`angle = W_p · x_i`, a projection of the token activations, like CARoPE/CoPE); (3) because the rotations are now data-dependent *and* SO(d_c) is non-commutative for `d_c>=3`, you cannot fold them into a closed-form `R^i` - you must form the running product `R_1 R_2 ... R_i` with a matrix-valued associative scan (the repo's `mru_scans.bk_mru.cummatprod`; "mru" = matrix recurrent unit). Relative position survives because the shared prefix product is orthogonal and cancels (`R R^T = I`), leaving `q_i (∏_{h=i..j+1} R_h) k_j^T`.
      **Why it is rejected on the position axis.** Two reasons. First, the *positional* version of this idea is already considered preempted in this roadmap - see "Context-conditioned curvature (prismatic RoPE)" below, which credits **PaTH** ([arxiv 2505.16381](https://arxiv.org/abs/2505.16381)) and **GRAPE** ([arxiv 2512.07805](https://arxiv.org/abs/2512.07805)) with the "content-dependent cumulative-rate RoPE warp and its matrix generalization." HDD-RoPE is a less-polished point in that same slot. Second, the cost is structural, not incidental: a per-token batched `torch.linalg.solve` (the Cayley inverse) plus an `O(seq · d_c^3)` matrix scan, with no fp16/bf16/tensor-core support (the author's own headline caveat is "significantly slower than xPos or RoPE"). Given [[project_calm_direction]] (the blocker is training *time*, not viability) and [[feedback_dependencies]], paying that in every attention layer's positional path for the repo's marginal, single-run TinyStories convergence bump - whose own author flags that the *absolute*-position variant converged *faster* and he cannot reproduce why, i.e. the effect may be regularizing noise, not the relative-position machinery - is the wrong trade. And **ArcHoPE does not already cover this** ([[project_attention_refactor]], `praxis/encoding/archope.py`): ArcHoPE's Serpent warp is *learnable* (per-band `alpha/beta/gamma/lambda/rho` trained once then fixed; `_position_freqs` is a pure function of `(position, depth)`) but **not data-dependent** - no token activations enter the angle. It relaxed strict relative-position (nonlinear in absolute `p`); it never became content-conditional. The `angle = W_p · x_i` move is exactly the piece ArcHoPE lacks.
      **The structural idea worth keeping, and its real home.** Strip away "position" and what remains is: *a data-dependent, non-abelian, orthogonal (norm-preserving, energy-conserving) linear recurrence* - a hidden state advanced by a content-conditioned element of SO(d). That is a **Koopman operator with a learned, input-conditioned generator** (eigenvalues pinned to the unit circle, energy conserved), which is precisely the object [[project_harmonic_latent_koopman]] is about (`research/framing.tex`, [harmonic_koopman.md](harmonic_koopman.md)) - except the harmonic thesis *freezes* the eigenbasis as constitutive, while this *learns and conditions* the transition. The bias on offer: relative "distance" between two tokens stops being the scalar `i-j` and becomes the ordered product of the intervening rotations - a path through the group, so token *content and order* shape the relative signal in a way a scalar offset structurally cannot. That belongs in the **memory / linear-recurrence track**, not the encoding registry: `praxis/memory/neural_memory.py` already maintains a data-dependent state with a parallel scan and data-dependent gates (`to_lr/to_momentum/to_decay`), and [[project_titans_memory]]'s `NeuralMemory` is the natural seat for an *orthogonal data-dependent transition* as a new `MEMORY_REGISTRY` profile - the energy-conserving counterpart to the current decaying-MLP memory. It also rhymes with the belief-state-as-Koopman-stepper item above (NextLat's transition MLP -> a harmonic/Koopman operator): both want a learned operator that steps a latent forward; this one constrains that operator to be orthogonal.
      **Cheapest first probe (cheap-then-structural).** (a) The ~5%-cost positional taste, if wanted at all: make ArcHoPE's phase genuinely *data-dependent* (CARoPE-style `delta = proj(x_i)` into the band rate) - abelian, so no scan and no per-token solve - and check whether content-conditioning the clock buys anything over the learned-but-static warp; this is a clean test of "does data-dependence matter here" without the non-abelian machinery. (b) The structural version: a `neural_memory` profile whose transition is a small data-dependent orthogonal map (Cayley or a few Givens rotations on a *small* `d_c`, scanned), A/B'd against `mal`/`mag` on long-context recall, watching whether an energy-conserving transition holds state longer than the decaying update. **Honest risks:** the non-abelian scan is expensive even off the position axis (keep `d_c` tiny, or approximate with commuting/Givens blocks); orthogonal transitions conserve energy but can *also* refuse to forget (no contraction = no natural decay, the opposite failure from Titans' weight-decay); and the whole salvage rests on the bet that the *recurrence structure*, not the *position framing*, is where the value was - unproven, since the repo only ever tested it as RoPE. Companion to the prismatic-RoPE, belief-state-Koopman, and Titans-memory items; substrate is `praxis/memory/`.

- [x] **Learnable RoPE theta, per-depth in ArcAttention**
      The default `theta=10000` is inherited from the original RoPE paper and almost certainly wrong at our scale; nobody has empirically tuned it for tiny models. Prior art on making it learnable: [IEEE Access 2025](https://ieeexplore.ieee.org/document/11084811) reports modest-but-consistent gains with four stabilizing tricks (separate LR, layer-wise init, cosine anneal, sigmoid bound) on Tiny Shakespeare / WikiText-103 / IWSLT'14; a [Frontiers 2025 study](https://www.frontiersin.org/journals/computer-science/articles/10.3389/fcomp.2025.1626899/full) sweeps fixed theta at character-level which is directly relevant to our regime. Genuinely Praxis-shaped angle nobody else has the substrate for: **per-recurrent-depth theta in ArcAttention**, parameterized as a `nn.Embedding(depth, 1)` of log-theta deltas on top of a learned base, so each pass through the recurrence sees a different RoPE base (coarse-to-fine positional zoom across depths). Caveat: theta interacts with HoPE's cutoff (`inv_freq = 1/theta^(2i/d)`), so the simplest combination freezes HoPE's `pos_dim` at init from the base theta and lets the per-depth deltas only shift band magnitudes, not the rotated/unrotated split.
      Done, on by default for every RoPE-derived encoding (`rope`, `hope`). `theta` is now a learned scalar on the shared `RoPE` base, parameterized in a bounded log space - `theta = THETA_MIN·(THETA_MAX/THETA_MIN)^sigmoid(z_base + z_delta[depth])`, range `[100, 1e6]`. Grounded in Huang & Chen (IEEE Access 2025, PDF read in full): they parameterize a per-layer multiplier `α^(l)` and benchmark four _optimization strategies_ (not four encodings), all applied together. Their ablation ranks separate-LR > sigmoid-constraint > cosine-anneal > layer-wise-init, and their sensitivity study finds base theta 400-800 optimal at small scale (not 10000) with `θ_min` best at 50-200. We adopt: the **sigmoid constraint** (their 2nd-strongest, and the only pure-parameterization strategy - fits the no-tuning rule), `THETA_MIN=100` (in their 50-200 band), and init at `THETA_INIT=500` (in their 400-800 band) rather than the legacy 10000. `z_base` inits theta to 500; `z_delta` is a zero-init `nn.Embedding(depth, 1)` so every depth starts there and specializes. Each recurrent-depth pass selects its delta via `current_depth`, threaded through `before_scores -> _compute_rope_embeddings -> _compute_inv_freq` and passed from all five call sites (causal, infini, arc, modular, syntaxes). HoPE overrides `_compute_inv_freq` to freeze its kept-band indices from the base theta on first call, so deltas only shift magnitudes. `inv_freq` is no longer cached (learned theta carries gradient, recomputed each forward).
      Deliberately skipped from the paper's four: **separate LR** (their most stability-critical strategy, but needs trainer/optimizer plumbing) and **cosine annealing** (a schedule); both are optimizer knobs that conflict with the no-tuning rule. We rely on the sigmoid bound + SandwichNorm for stability instead - revisit separate-LR (as a fixed `base_lr/100` ratio, which the rule would permit) only if a real run shows theta destabilizing. Also chose a _uniform_ zero-init delta over the paper's gradient layer-wise init, so depths specialize purely from data. Deferred: dashboard metrics for the learned theta / per-depth spread (no collector + JS wiring yet, mirroring how Wavelet deferred its metrics).

- [x] **Surprise-triggered (event-aligned) memory chunking for Titans**
      Replace the Titans memory's fixed `chunk_size` with variable-length chunks cut at surprise spikes, borrowing the event-segmentation idea from EM-LLM ([Fountas et al. 2024](https://arxiv.org/abs/2407.09450)). Both methods already key on prediction-error "surprise"; EM-LLM uses it to place episodic-event boundaries, while our memory already computes a per-chunk surprise - so the step is to let a surprise threshold (e.g. `μ + γσ` over a moving window) trigger the update boundary instead of a fixed stride. Goal: align memory writes to coherent events rather than arbitrary token windows. Pairs with the energy/Adam update work in the Titans memory module.
      Done: energy mode (`mal_energy`) runs the update on a `segment_block`=16 grid; blocks merge into events until a causal `μ + γσ` (γ=1) surprise spike or the `chunk_size` cap, whichever comes first. Boundaries reset the Adam EMAs (weights persist) via the existing parallel scan - no sequential loop on the hot path. Diagnostics: Memory Event Size (mean/min/max). Next: optional graph-theoretic boundary refinement (EM-LLM's secondary step), and per-event W0 re-freeze if the momentum-reset approximation proves too coarse.

- [x] **ArcHoPE: per-feature Serpent warp on the rotary phase (`encoding: arc`)**

Lives in `praxis/encoding/archope.py`, opt-in via `arc` in `ENCODING_REGISTRY` (`hope` stays default), built on RoPE's `_position_freqs` seam (no change for `rope`/`hope`). Accepted trade-off: the active warp makes phase depend on absolute position, relaxing strict relative-position (the CARoPE/PaTH trade). Deferred: per-band specialization dashboard metrics. **Status: shipped and driving CALM ablations today (`experiments/calm-d.yml`).**
