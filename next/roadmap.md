# Praxis Roadmap

Tracked work items for the Praxis research framework. These range from near-term wins to longer-term architectural goals. Items are roughly ordered by priority, not dependency.

---

- [ ] **Scheduled sampling at training time**
      Train the model on its own predictions for a fraction of positions: instead of always feeding the ground-truth previous token, occasionally substitute the model's sampled token. This closes the train/inference distribution gap that drives exposure bias (Bengio et al. 2015). Hook in at the trainer's input-id construction step; expose the schedule (start at 0%, ramp to ~25% over training) as a CLI knob. Natural A/B against the existing `contrastive_token` loss, which targets the same failure mode from a different angle.

- [ ] **Finish retraining tokenizers at varied sizes**
      Complete the in-progress work of retraining the project's tokenizers across a range of vocabulary sizes so each model scale has a tokenizer that actually fits it, rather than reusing a single vocab across small, medium, and large configurations. Varied sizes let us study the interaction between vocab size, embedding parameter count, and downstream loss without confounding it against the tokenizer's training corpus. Tracked in [0-5788719150923125/praxis#50](https://github.com/0-5788719150923125/praxis/issues/50).

- [ ] **Fix metrics loading performance for long runs**
      As training progresses, the `/api/metrics` endpoint gets progressively slower because it re-reads and re-downsamples (LTTB) the full metrics history from SQLite on every request. Investigate server-side caching of the downsampled response (invalidated on new writes), pre-materialized summary tables, or incremental approaches where only new rows since the last request need processing. The frontend's ETag/304 caching helps with repeated identical requests but does nothing for the first fetch after new data arrives.

- [ ] **Integrate Arxiv API as a training data source**
      Add an Arxiv-backed dataset that pulls papers via the public Arxiv API (`http://export.arxiv.org/api/query`) and streams abstracts (and optionally full-text PDFs, parsed to plain text) into the training pipeline. Register it as a new `type` in `DATASETS` (`praxis/data/config.py`) so it can drop into any collection alongside the existing HuggingFace and directory sources. Respect Arxiv's rate limits (~1 request / 3 seconds) with on-disk caching so re-runs do not re-hit the API, and expose category filters (e.g. `cs.LG`, `cs.CL`) as dataset config so we can target research-relevant subsets.

- [ ] **Propagate task-weighted loss into layer-wise training**
      The new `loss_weights` channel is plumbed through `praxis.losses.cross_entropy`, `focal`, `mile`, `stablemax`, `halo`, and `contrastive_token` via `weighted_reduce`, but `praxis/losses/layer_wise.py` and the layer-wise / MonoForward training paths still call the criterion without forwarding `loss_weights`. Update `compute_layer_wise_loss()` to accept and pass through the per-token weights, and have the trainer thread `task_type_ids` / `assistant_mask` into each layer's local objective. Until then, layer-wise runs see uniform weighting regardless of `--task-weights` or `--mask-prompts`.

- [ ] **Table-free projection embeddings (SGNN / PRADO style)**
      Prototype an embedding that *computes* token vectors from a fixed hash/LSH projection of byte n-grams instead of retrieving them from a `vocab x dim` table, following SGNN ([Ravi & Kozareva 2018](https://aclanthology.org/D18-1105/)) and PRADO / pQRNN. The fixed projection carries zero learnable params and leans on Johnson-Lindenstrauss to preserve distances, with a small learned layer downstream recovering features - trading embedding params/VRAM for a bit more compute. Fits as a new `EMBEDDING_REGISTRY` entry, and pairs naturally with the byte-latent stack, which already hash-embeds n-grams.

- [ ] **Cyclic Mixture-of-Depths routing (ping -> pang -> pong -> ping)**
      The current MoD routers route a fraction of tokens through each layer in the forward direction only. A cyclic variant would let tokens revisit earlier layers, turning the depth dimension from a chain into a graph - the substrate for "hard tokens loop back through heavy layers while easy tokens take the short path." Cheapest first probe: add a new entry to `ROUTER_REGISTRY` that, after a layer's normal selection, sends some fraction of the surviving tokens back to a randomly-chosen earlier layer. Measure whether the cyclic paths concentrate on high-entropy positions, code, or structured output.

- [ ] **Auto-derive each run's math expression from its compiled architecture**
      Have model classes carry a symbolic description of their own operation (SymPy, einops-style strings, or a small custom DSL), then traverse the compiled architecture and compose those fragments into a single concise formula for the full forward pass. The output lives next to every run's logs so "the math" is always inspectable and citable alongside the loss curves. Open questions: how to handle data-dependent control flow (MoD routing, halting), and whether per-module math is declared by hand or extracted from the implementation via tracing. Related prior art worth scanning first: Penzai (Google's pedagogical NN library), Equinox/Flax module reprs, SymPy lazy simplification.

- [ ] **Fractional-seed init sweeps for ablation sensitivity**
      Generalize the integer `--seed` to a float, where the fractional part interpolates the initial weights between two adjacent integer seeds (e.g. `3.14` blends 14% of `seed=3` weights with 86% of `seed=4`). Closest known analogue is the variation-seed / subseed-strength mechanic in Stable Diffusion, applied to noise rather than weights. The use case here is a cheap 1D walk through init space: sweep the fraction during an ablation to measure how smoothly a metric varies with init, rather than getting a single noisy point per integer seed. Note that for i.i.d. Gaussian inits, naive lerp shrinks variance at α=0.5, so the blend should rescale (or use slerp) to keep the init distribution well-behaved.

- [ ] **Harden integration error handling**
      The integration loader (`praxis/integrations/loader.py`) and individual integrations silently swallow exceptions at every level - spec discovery, module loading, hook execution, and runtime errors all land in bare `except Exception: pass` or quiet `print` calls. This hides real bugs (e.g. a stale `from praxis import api` import went unnoticed because the Discord integration caught the `ImportError` and printed a one-liner). Integrations should fail loudly by default, distinguishing between import/setup errors (which are always bugs and should raise) and runtime errors (which may warrant graceful degradation). Consider a `critical` flag in `spec.yaml` so truly optional integrations can degrade without stopping training, while misconfigured ones surface immediately.

- [ ] **Experiment with Backpack-style multi-sense embeddings**
      Prototype a Backpack Language Model ([Hewitt et al. 2023](https://arxiv.org/abs/2305.16765)) embedding layer, where each token gets *k* learned "sense vectors" rather than a single embedding, and the model produces contextual output as a weighted sum over the senses of tokens in the sequence. The appeal is interpretability and compositionality: senses in the original paper specialized in ways that were human-legible (syntactic role, lexical content, etc.) and could be intervened on directly. Fits naturally as a new entry in `EMBEDDING_REGISTRY` so it can be swapped against the default embedding without touching the rest of the stack. Worth testing whether sense specialization survives in our small-scale regime, and whether it composes with Prismatic / per-depth bias experiments.

- [ ] **Explore neuro-symbolic path prediction with branchless execution**
      Prototype a small neuro-symbolic core where the "program" is an explicit symbolic state machine (registers, ops, routing table) and tiny neural networks act as differentiable predicates that steer which path executes. Borrow the branchless-programming idiom: every branch becomes a soft-weighted selection over precomputed alternatives, keeping the whole system differentiable end-to-end while retaining inspectable symbolic state. Related prior art worth skimming first: Neural Turing Machines / DNC (Graves et al.), DreamCoder (Ellis et al.), AlphaGeometry's neural-guided symbolic search. The research question is whether a model like this can exceed vanilla-transformer expressivity (TC0) at small scale without relying on chain-of-thought as the sole Turing-completeness escape hatch.

- [ ] **(historical) CALM produces noise: audit vs shaochenze/calm (2026-05-30)**
      The port is faithful - VAE (free-bits KL), embed_proj patch input, energy head, energy_score (2·E‖X-Y‖ - E‖X-X'‖ over M=100 posterior samples), and patch-vote temperature generation all match the reference. The deviation is protocol, not math. Reference trains the autoencoder as a fully separate run (~30k steps / 15B tokens) to near-perfect reconstruction, saves it, then loads it frozen for CALM. Ours collapses both into one run with `ae_freeze_steps == warmup_steps`, so the codec freezes long before it reconstructs - stage 2 then learns to predict latents that decode to garbage. Direction: (1) verify codec fidelity first - decode posterior samples of real text and confirm it reconstructs (watch `val_codec_bpb`); (2) decouple stage 1 into its own checkpointed AE run, don't tie the freeze to LR warmup; (3) only after the AE is near-lossless, train the energy LM, and bump `energy_samples_n` (8 is a noisy repulsion estimate). Scale also matters: reference is 371M-1.8B params; tiny undertrained models may not reach fluency regardless.

- [ ] **Unify registry pattern with a lightweight `Registry` class**
      Replace the 43 ad-hoc `*_REGISTRY` dicts (currently spread across the tree) with a thin `Registry` subclass of `dict` that standardizes error messages, supports optional decorator-based registration, and provides a consistent typed interface. Migration should be incremental - the class is still a dict, so existing `REGISTRY.keys()` and `REGISTRY[name](config)` call sites stay unchanged. Also audit `SAMPLER_REGISTRY` (maps strings to strings) and `EMBEDDING_REGISTRY` (maps every block type to the same class) as candidates for removal.
      Stub landed: a `praxis.registry` namespace now exists as the home for this work, with `discover_registries()` / `count_registries()` as its first piece - an AST scan that finds every module-level `*_REGISTRY` definition (43 today). The auto-docs/README headline now reports that real number instead of a hand-maintained "~20". Renamed `NON_REGISTRY_PACKAGES` -> `INFRASTRUCTURE_PACKAGES` so no non-registry constant collides with a plain `*_REGISTRY` match. Next step toward the actual unification: define the `Registry` type in `praxis.registry` and migrate the first dict to it.

- [ ] **Preference-pair objective for hh-rlhf (and a real use for `rejected`)**
      `Anthropic/hh-rlhf` joined the `focused` collection (2026-06-10), but only SFT-style: the `chosen` transcripts are parsed into chat messages (`DataFormat.HUMAN_ASSISTANT`) and the `rejected` column is dropped on the floor. The signal that makes the dataset valuable is the *pair*. None of the existing RL policies fit: REINFORCE/GRPO need a reward channel and CoT is supervised - nothing in `RL_POLICIES_REGISTRY` consumes pairwise preferences. The natural first move is an offline preference loss, not online RL: DPO (Rafailov et al. 2023) needs a frozen reference model (awkward in a single-run framework), so reference-free variants - ORPO (odds-ratio penalty folded into SFT) or SimPO (length-normalized margin) - fit the one-model-one-run shape better. Work items: (1) a batch path that emits chosen/rejected as paired sequences sharing a prompt (the builder currently has no concept of a paired sample); (2) the loss as a registry entry alongside the existing criteria, gated to pairs-bearing batches; (3) decide its interaction with the assistant mask - preference applies to assistant turns only. Until then, hh-rlhf is just more conversation data.

- [ ] **Learned / dynamic per-token temperature**
      Explore having the model predict its own sampling temperature alongside its token logits, rather than treating temperature as a fixed inference-time hyperparameter. The intuition: the model is the entity best positioned to know when it is confident (use low temperature) vs. exploring (use higher temperature). Implementation sketch: a small auxiliary head off the final hidden state that emits a scalar (or short vector) controlling the softmax temperature for the *next* token's distribution, trained jointly with NLL. Related prior art: Mirostat (Basu et al. 2020) for dynamic-temperature sampling without a learned predictor, and "Hot or Cold? Adaptive Temperature Sampling for Code Generation with Large Language Models" ([Zhu et al. 2023, Peking University](https://arxiv.org/abs/2309.02772)), which adapts temperature by token type via a heuristic (high for "challenging" tokens, low for "confident" ones) rather than a learned predictor. NB: the link above resolves to the Zhu et al. paper, *not* an Apple work - the separately-recalled "Apple paper on learned per-token temperature" remains unidentified, if it exists at all. Worth pairing naturally with the region-conditional sampling item above.

- [ ] **Adopt a proper namespaced logging interface (retire scattered `print()`)**
      The codebase emits a lot of status through bare `print()`. Move to per-subsystem loggers - `logging.getLogger("praxis.optim")`, `"praxis.data"`, `"praxis.encoder")`, etc. - so output carries a namespace + level instead of an undifferentiated stream. The plumbing already favors this: the dashboard installs a root handler (`DashboardStreamHandler`) and formats records as `name - LEVEL - message`, so namespaced loggers flow into the LOG panel with structure for free, while `print()` only survives because `start()` also redirects stdout. Wins: filterable/levelable output, per-module verbosity, clean separation of user-facing CLI text (keep as `print`) from diagnostic logging, and log lines that read like the `praxis.optim - INFO - ...` examples in the generated `terminal.webp` capture. Sketch: define a `praxis.logging` helper that hands back a namespaced logger (and sets sane defaults/levels), sweep `print()` calls that are really diagnostics over to it, and leave genuine CLI/UI output alone. Watch the interaction with the dashboard's aggressive logger reconfiguration in `TerminalDashboard.__init__` (it retrofits every existing logger) - the convention and that retrofit should be designed together.

- [ ] **Unified knowledge base: search / index / spider on the Gymnasium tab (2026-06-01)**
      A shared human/assistant knowledge base on the Gymnasium tab's `Read` / `Evaluate` toggles. **Read** turns the text box into ranked search over an index (a browsable, self-updating wiki); **Evaluate** lets the assistant search the KB mid-conversation - and toggling `Read` off revokes that access live, the load-bearing idea (scope as a revocable capability, not static config). The corpus is mostly already on disk: run history (`build/runs/<hash>/`), the auto-generated `docs/` wiki (never drifts from code), and external Praxis-related docs pulled by a spider; FTS5-first, with semantic/vector search deferred behind it per the dependency rule. Terminology flag carried forward: the "differential privacy" framing is really access-control / capability-scoping + region encryption, not DP - decide which property is the actual goal.
      **Status: POC well underway.** Shipped: the FTS5 KB index (`praxis/kb/`), the `praxis.spider` crawler (`build/spider.db`), the `/api/kb_search` route + Gymnasium `Read` wiring, a `kb` dataset source (`praxis/data/datasets/kb.py`, so the corpus doubles as training data), and the `tools/index_kb.py` ingestor. Remaining: Evaluate-mode agentic access, the access-control / encryption layer, and (deferred) semantic search.

- [ ] **Homeostatic engagement reward: the user's attention as a self-driven signal**
      A `Print` stage on the Gymnasium button row (after `Read`/`Evaluate`): the model emits a single question, the user engages, and that feeds a slowly-replenishing "energy" variable the model is optimized to keep near a setpoint - diminishing returns and a multi-hour decay, like hunger or attention (homeostatic RL, Keramati & Gutkin 2014). The satiation term is the good part: self-limiting, kills reward-spam, and mirrors the EMA-integrated-return design already in `harmonic_weight_rl`. The crux is the reward *choice*: **"questions the user answers correctly" (calibration / active learning) - NOT "maximize engagement," which is the recommender-system path to clickbait and manipulation.** Practical wall: human reward is low-volume, so scope it to tuning a small controller's narrow behavior (not a generation policy from scratch), and use a simulated user for sample density during development; an offline reward model would reintroduce proxy-gaming. Staging, smallest-first: (1) `Print` logs `(question, engaged?, correct?)` as data; (2) a homeostatic energy reward feeding a small controller, the real work being the async UI-event -> controller reward channel; (3) a `BrierLM`-style eval of the question-posing behavior. Kill criterion: if it ever optimizes attention over usefulness, stop. Full write-up: [homeostatic_engagement.md](homeostatic_engagement.md).

- [ ] **Procedurally generate per-run research reports (LaTeX -> PDF)**
      Compile a structured paper from each run the way a build system compiles a binary: the source is the experiment config, the compiled architecture, and the empirical results; the output is a LaTeX document (PDF-exportable, optionally via a button on the web dashboard). Explicitly *not* an LLM-written prose paper - no generated narrative, no hallucinated claims. Every report follows one fixed template and looks near-identical by design; the value is the accuracy and reproducibility of the numbers, not the wording. Readers trust it precisely because it is mechanical. Sections fill from real artifacts: the auto-derived forward-pass formula from the architecture-math item above, a config / hyperparameter table, metric curves and final values pulled from the run databases (`metrics.db`, `dynamics.db`), ablation deltas across seeds and variants, and a bibliography assembled from the prior-art citations already attached to each module / registry. Think literate-programming and reproducible-research toolchains (Quarto, knitr / R Markdown, Pweave, papermill, Distill) rather than a chatbot - reports are rendered from source and results, not authored. Open questions: where citations live as structured data (per-module BibTeX vs. a central registry), how to diff two reports for an A/B comparison, and whether rendering runs offline as a `tools/` utility or on-demand behind a dashboard export endpoint.

- [ ] **2D optimizer-trajectory visualization (real-data Rastrigin/Rosenbrock-style path)**
      pytorch-optimizer ships parametric 2D trajectory plots of each optimizer descending toy loss surfaces ([visualization gallery](https://pytorch-optimizers.readthedocs.io/en/latest/visualization/)). We want the same *picture* but on real data: project the high-dim parameter vector to 2D and draw the optimizer's path over training. The scalar half of optimizer telemetry shipped (`praxis/metrics/optimizer.py`: the "optimizer" dynamics card group - `opt_update_rms`, `opt_update_weight_ratio`, `opt_momentum_grad_cos`, `opt_second_moment_rms`, schedule-free `opt_sf_spread` + `opt_gate_mean`); this item is the deferred non-scalar viz. Design: a *fixed random 2D projection* of the flattened params (two frozen random unit vectors -> two dot products per snapshot, cheap and shape-stable), plotting **both** the schedule-free iterate `z` and the running average `x` as separate paths - so you watch the iterate explore while the average smooths behind it (the bias-variance story as a real-data trajectory). Reuse the existing `harmonic_curve` 2D-parametric-path renderer. Plumbing needed (the reason it's deferred): the web layer has no optimizer handle - `/api/head_snapshots` reads `model.head.dashboard_snapshots()` - so add a `/api/optimizer_snapshots` route fed by a stashed optimizer (or have the `DynamicsLogger` push trajectory points to a small ring buffer the route serves), a frontend renderer entry, and a `praxis/web/src` -> `build.py` rebuild. Per-wrapper paths (plain vs gated vs wave schedule-free) would make a clean visual ablation.

- [ ] **$SLEEP token: shared checkpoint store, resume from where peers left off**
      Speculative. If Praxis is a standardized framework, two peers running the same `experiment.yml` against the same data should be producing compatible checkpoints. A shared store (IPFS-pinned, or a simple HTTP index to start) would let a fresh `--alpha` run fetch a peer's checkpoint that already has 20B tokens of training behind it, instead of starting from random init. The hard parts are the equivalence check (config hash + dataset manifest + tokenizer hash + Praxis version + seed lineage) and Sybil-resistant validation that a posted checkpoint actually reached the claimed loss. The $SLEEP token would reward peers who pin checkpoints other people actually resume from. Thematic fit: caching state means the rest of the network doesn't have to wake up to recompute it. Worth shelving until the deterministic-equivalence story is solid - without that, the index is a junkyard.
      `--phi`, `--pile`, `--no-source`, and the `minimal_data` feature flag are gone; selection now flows through two CLI list flags, `--train-datasets` and `--validation-datasets`, each accepting space- or comma-separated collection names (e.g. `--train-datasets base phi`, `--train-datasets base,phi`). Collections live in `DATASET_COLLECTIONS` in `praxis/data/config.py`; individual entries in `DATASETS` carry a `type` field (`huggingface`, `directory`, `synthetic-tool-calling`) so that what used to be hard-coded branches - the `./` source tree via the old `"self"` format, the synthetic tool-calling gate keyed on `"phi" in datasets`, and the auto-`dev` swap under `minimal_data` - are now just typed dataset entries dropped into whichever collection you want. `environments/dev.yml` sets `train_datasets: [dev]` directly; the 12 experiment YAMLs that previously set `phi: true` now set `train_datasets: [base, phi]`. Individual datasets are addressable as collection members today; bare-dataset selection via the CLI (without a collection wrapper) remains a future-nice-to-have.

- [ ] **Elimination tournament over the swarm: last expert standing as consensus**
      A selection rule for the swarm-dynamics item above, answering its open contribute-back question with a knockout instead of an average. Run elimination rounds over the experts - sample, compare on the current batch, drop the loser(s) - and keep going until exactly one survives. That survivor is the consensus pick (its output is taken as the "truth" for the batch); read it honestly as *the elected expert*, not ground truth. The twist is an inverted learning signal: **the winner is frozen this step (it already speaks for the batch), and the gradient goes to the experts that were eliminated** - they're the ones who need to improve, while the leader rests. This is the anti-rich-get-richer move: winner-rests gives the rest room to catch up and rotates leadership as the frozen champion gets overtaken, keeping the population diverse rather than collapsing onto one expert. It's also a knockout variant of the count-based vote Praxis already uses (CALM patch-vote, the swarm's stratified vote) - single winner by elimination rather than a weighted tally.
      Home is the same `--orchestration-type` profile + `MIXING_REGISTRY` seam as the swarm item; substrate is `praxis/orchestration/`. Honest open questions: the elimination criterion (pairwise held-out loss? agreement? confidence?); what the losers train *toward* - their own labels (normal SGD with the leader sitting out) vs. distilling to the winner's output (faster convergence but a mode-collapse risk, the very thing winner-rests is meant to avoid); whether freezing the winner actually keeps the population healthy or just slows the leader; and the cost of a full bracket per step over thousands of experts - mitigated by the same stratified/`log N` sampling the sibling item uses (a sampled bracket, not an exhaustive one). Cheapest first probe: 8-32 experts, single-elimination by held-out batch loss, freeze the winner and step only the losers for that batch, and check whether best-expert quality *and* population diversity beat training everyone every step. Companion to the swarm-dynamics item and [world_models.md](world_models.md).

- [ ] **Edit-trajectory training: `[REMOVE]`/`[ADD]` markers as multi-timescale time**
      Teach the model the *process* of a revision, not just its endpoints, by training the same rolling-context batch across staged passes that spell out an edit. The kernel is **2 decisions, 3 steps**: (1) replace a span with `[REMOVE]`, train the batch; (2) add an `[ADD]` marker while leaving `[REMOVE]` in place, train the batch again; (3) actually delete the `[REMOVE]` text, train the batch a third time - done. The two markers are explicit edit operations in the token stream, and the three passes give the model the *trajectory* (mark -> stage -> apply) rather than only before/after. Read it as a second, coarser time axis layered on token order: tokens are the fast clock, the staged edit markers a slow one - "backpropagation through epochs," learning signal threaded across passes over the same content. Closest established cousins are edit-based / text-editing models (Levenshtein Transformer, LaserTagger) and diffusion LMs that learn a corruption→denoise *trajectory*; this differs in making the trajectory an explicit, marked, staged training schedule.
      The speculative core - and the reason it's hard to reason about - is that the decision space here is **timeless**. *Which* tokens to `[REMOVE]` and *what/where* to `[ADD]` is an order-free set of operations; the 3-step staging *injects* a synthetic time onto it. The strangeness ("a thing like ours without time in it makes no sense") is the point: time isn't inherent to the edit, it's constructed in the data, and that's exactly the structure we'd be teaching. Honest caveats, since this is underspecified: decide the loss target at each pass (predict the marked span? the applied result?); whether it's curriculum vs. data augmentation; whether 3x-training one batch overfits; and how the 2-decision/3-step kernel generalizes to many simultaneous edits. Cheapest first probe: add `[REMOVE]`/`[ADD]` special tokens, write a synthetic generator that emits the three staged sequences from a source span (sibling to the print synthetic-samples + tasker work), train on it, and check whether the model learns to honor `[REMOVE]` and generate plausible `[ADD]` content - i.e. to *apply an edit*, not just describe one.

- [ ] **Spectral temperature: a fast-weights "harmonic twist" knob at sampling time**
      A second control axis alongside the usual (logit/vote) temperature: a test-time, fast-weights modulation of the harmonic machinery's *spectrum*, not its logits. The exact, defensible core - "compress amplitudes" = a **power law on the harmonic amplitude spectrum**, `|a_k| -> |a_k|^(1/T_h)`: identical math to softmax temperature but in the spectral domain. `T_h<1` sharpens (a few harmonics dominate -> more structured/peaky output), `T_h>1` flattens (uniform harmonics -> smoother). Cleanest home: a `set_spectral_temperature(T_h)` fast-weight on `HarmonicHead` that powers/scales the learned amplitudes before the field is rebuilt - cheap, reversible, no retraining, and the existing spectrum/heatmap dashboards would show it shift live. "Compress frequencies / twist" is the heavier phase-2 extension: a frequency-axis warp is a chirp / fractional-Fourier shear (frequency-dependent phase rotation - a genuine time-frequency "twist"); the field's frequencies come from a fixed IRFFT2 grid, so warping means recomputing it.
      Two honest caveats that scope this. (1) **It is a control axis, not a probabilistic temperature.** The harmonic head is deterministic given its input, so twisting its spectrum changes the decode deterministically (a sharpness/texture knob), it does not sharpen a sampling *distribution*. The one place a harmonic twist would truly be "part of sampling" is the **energy head's noise**: give that noise spectral structure and twist its spectrum at test time - the most novel angle, but the most speculative (the energy head currently draws flat uniform noise; structured noise must be invented first). (2) **Not a fix for current CALM incoherence, and could hurt.** In CALM the harmonic head lives in the *codec decode* path (latent -> VAE decode -> harmonic field x crystal -> tokens), which already reconstructs at ~100%; the failure is upstream (the energy head's conditional, see the CALM root-cause item). Twisting a near-lossless decoder at test time can only distort a decode that works - it can't recover what the generator never predicted. So this is a "once it generates" creative/control knob, to revisit after the energy-head conditional is learning real next-patches. Cheapest first probe (deferred until then): the amplitude-power knob on `HarmonicHead`, A/B'd as a generation-config override like `calm_noise_scale`, on a model that already produces fragments.

- [ ] **Swarm dynamics: population learning over thousands of 16x16 experts**
      The orchestration pool (`praxis/orchestration/`, observer mode landed) opens a *different learning regime* once it's not a handful of experts but **thousands of tiny (16x16, 1-layer) transformers**, each training detached on real batches and self-tracking cheap loss/accuracy EMAs (sampled, not polled). At that scale the swarm stops being a mixture and becomes a **population**, and population-based optimizers become the natural controller: **particle swarm optimization** (each expert a particle in weight/hyperparameter space, pulled toward personal-best and swarm-best) and **ant-colony**-style reinforcement (good experts lay "pheromone" that biases which regions of config space new spawns explore) are the obvious first two to try. Honest open questions: does population diversity actually beat a single tuned expert at the same total params (the `world_models.md` hinge); do PSO/ACO converge or thrash under non-stationary streamed data; and what's the right tier binning so the low-count vote is unbiased. - [x] **Learnable RoPE theta, per-depth in ArcAttention**
      The default `theta=10000` is inherited from the original RoPE paper and almost certainly wrong at our scale; nobody has empirically tuned it for tiny models. Prior art on making it learnable: [IEEE Access 2025](https://ieeexplore.ieee.org/document/11084811) reports modest-but-consistent gains with four stabilizing tricks (separate LR, layer-wise init, cosine anneal, sigmoid bound) on Tiny Shakespeare / WikiText-103 / IWSLT'14; a [Frontiers 2025 study](https://www.frontiersin.org/journals/computer-science/articles/10.3389/fcomp.2025.1626899/full) sweeps fixed theta at character-level which is directly relevant to our regime. Genuinely Praxis-shaped angle nobody else has the substrate for: **per-recurrent-depth theta in ArcAttention**, parameterized as a `nn.Embedding(depth, 1)` of log-theta deltas on top of a learned base, so each pass through the recurrence sees a different RoPE base (coarse-to-fine positional zoom across depths). Caveat: theta interacts with HoPE's cutoff (`inv_freq = 1/theta^(2i/d)`), so the simplest combination freezes HoPE's `pos_dim` at init from the base theta and lets the per-depth deltas only shift band magnitudes, not the rotated/unrotated split.
      Done, on by default for every RoPE-derived encoding (`rope`, `hope`). `theta` is now a learned scalar on the shared `RoPE` base, parameterized in a bounded log space - `theta = THETA_MIN·(THETA_MAX/THETA_MIN)^sigmoid(z_base + z_delta[depth])`, range `[100, 1e6]`. Grounded in Huang & Chen (IEEE Access 2025, PDF read in full): they parameterize a per-layer multiplier `α^(l)` and benchmark four *optimization strategies* (not four encodings), all applied together. Their ablation ranks separate-LR > sigmoid-constraint > cosine-anneal > layer-wise-init, and their sensitivity study finds base theta 400-800 optimal at small scale (not 10000) with `θ_min` best at 50-200. We adopt: the **sigmoid constraint** (their 2nd-strongest, and the only pure-parameterization strategy - fits the no-tuning rule), `THETA_MIN=100` (in their 50-200 band), and init at `THETA_INIT=500` (in their 400-800 band) rather than the legacy 10000. `z_base` inits theta to 500; `z_delta` is a zero-init `nn.Embedding(depth, 1)` so every depth starts there and specializes. Each recurrent-depth pass selects its delta via `current_depth`, threaded through `before_scores -> _compute_rope_embeddings -> _compute_inv_freq` and passed from all five call sites (causal, infini, arc, modular, syntaxes). HoPE overrides `_compute_inv_freq` to freeze its kept-band indices from the base theta on first call, so deltas only shift magnitudes. `inv_freq` is no longer cached (learned theta carries gradient, recomputed each forward).
      Deliberately skipped from the paper's four: **separate LR** (their most stability-critical strategy, but needs trainer/optimizer plumbing) and **cosine annealing** (a schedule); both are optimizer knobs that conflict with the no-tuning rule. We rely on the sigmoid bound + SandwichNorm for stability instead - revisit separate-LR (as a fixed `base_lr/100` ratio, which the rule would permit) only if a real run shows theta destabilizing. Also chose a *uniform* zero-init delta over the paper's gradient layer-wise init, so depths specialize purely from data. Deferred: dashboard metrics for the learned theta / per-depth spread (no collector + JS wiring yet, mirroring how Wavelet deferred its metrics).

FoX is score-side decay, not a position encoding.

- [x] **Environment variable support for CLI arguments**
      Every argparse flag (including dynamically added ones from integrations and experiment/environment discovery) is now settable via `PRAXIS_<DEST>` env vars. `EnvVarLoader` (`praxis/cli/loaders/env_vars.py`) iterates `parser._actions` after parsing, coerces the raw string to the action's declared type (bool-ish for store*true, type callable for scalars, comma or JSON list for `nargs`/`append`/`extend`, validated against `choices`), and applies it unless the CLI flag was given explicitly. Precedence top-down: explicit CLI > active environment YAML > `PRAXIS*\*` env var > experiment YAML > argparse default.
